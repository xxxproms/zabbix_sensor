# Zabbix: ключи агента, настройка в веб-интерфейсе и протокол

Документ и прошивка согласованы с пассивным протоколом **Zabbix agent** на **Zabbix 7.4.8** (server/proxy **сам подключается** к Arduino на порту **10050**). Ниже — **подробные шаги в GUI** и технические детали протокола.

Прошивка: [zabbix_ds18b20/zabbix_ds18b20.ino](zabbix_ds18b20/zabbix_ds18b20.ino) (порт **10050**). Монтаж датчиков: [MONTAZH_I_PAYKA_DS18B20.md](MONTAZH_I_PAYKA_DS18B20.md). Схемы и BOM: [SCHEMA_I_SOEDINENIYA.md](SCHEMA_I_SOEDINENIYA.md), [OBSHIY_BOM.md](OBSHIY_BOM.md). Общий чек-лист: [NASTR_AYKA_I_OTLADKA.md](NASTR_AYKA_I_OTLADKA.md).

## Ключи на стороне Arduino (пассивные проверки)

| Ключ | Назначение |
|------|------------|
| `agent.ping` | Доступность агента (ответ `1`) |
| `env.temp` | Температура, датчик 1 (ROM `tempSensor1` в скетче) |
| `env.temp1` | Датчик 2 |
| `env.temp2` | Датчик 3 |
| `env.temp3` | Датчик 4 |

Любой другой ключ → тело ответа `ZBX_NOTSUPPORTED` с корректным заголовком `ZBXD`.

**Важно:** ключ вводится **без** префикса `sensor[` и **точно** как в таблице (регистр, точки — как в прошивке).

## Подробно: как добавить датчики в Zabbix 7.4.x (веб-интерфейс)

Названия меню ниже соответствуют **английскому** интерфейсу Zabbix; в **русской** локализации обычно: **Сбор данных** вместо *Data collection*, **Узлы сети** вместо *Hosts*, **Элементы данных** вместо *Items*.

### Шаг 1. Сеть и доступ

1. У Arduino в сети должен быть **стабильный IP** (у вас сегмент **10.10.0.0/23**, см. скетч).
2. С **сервера Zabbix** или **прокси**, который будет опрашивать хост, должен быть **доступен TCP-порт 10050** до IP Arduino (маршрутизация, ACL, без «закрытого» порта на промежуточных фаерволах). Опрос **пассивный**: инициатор — **Zabbix**, не Arduino.

### Шаг 2. Группа узлов (по желанию)

1. **Data collection** → **Host groups** → **Create host group**.
2. Имя, например: `Arduino / DS18B20` → **Add**.

Группу можно не создавать и выбрать существующую (например `Linux servers`), но отдельная группа упорядочивает учёт датчиков.

### Шаг 3. Создание хоста и интерфейса Agent

1. **Data collection** → **Hosts** → **Create host**.
2. **Host name** — уникальное имя, например `sensor-rack-01` (латиница, без пробелов или по правилам вашей политики имён).
3. **Groups** — добавьте группу из шага 2 (или другую).
4. Блок **Interfaces** → **Add** → тип **Agent**:
   - **IP address** — IP Arduino (например `10.10.0.50`).
   - **DNS name** — можно оставить пустым, если используете только IP.
   - **Connect to** — **IP** (если заполняли IP).
   - **Port** — **`10050`** (как в прошивке).
5. **Add** интерфейса, затем внизу страницы **Add** (сохранить хост).

Пока **без** элементов данных хост будет «пустым» — это нормально.

### Шаг 4. Элементы данных (температура и пинг)

Для **каждого** нужного ключа создаётся отдельный элемент.

1. Откройте **Data collection** → **Hosts** → щёлкните **имя хоста** → вкладка **Items** → **Create item**.

Заполните для **первой температуры** (датчик 1):

| Поле | Значение |
|------|----------|
| **Name** | Понятное имя, например `Temp sensor 1 (env.temp)` |
| **Type** | **Zabbix agent** |
| **Key** | `env.temp` |
| **Type of information** | **Numeric (float)** |
| **Units** | `°C` (или `C`) |
| **Update interval** | например `1m` (чаще — больше нагрузки на Arduino и сеть) |

Сохраните (**Add**).

2. Повторите **Create item** для остальных датчиков:

| Key | Пример Name |
|-----|----------------|
| `env.temp1` | Temp sensor 2 |
| `env.temp2` | Temp sensor 3 |
| `env.temp3` | Temp sensor 4 |

3. Для **доступности** создайте элемент с ключом **`agent.ping`**, тип **Numeric (unsigned)** или **Text** (значение будет `1`), интервал можно **1m** или реже.

### Шаг 5. Проверка, что данные приходят

1. **Monitoring** → **Latest data** — выберите хост, убедитесь, что элементы в состоянии **Supported** и видны значения.
2. Если **Unsupported** — откройте элемент → **Info** / сообщение об ошибке; частые причины: неверный IP, порт не 10050, фаервол, Arduino выключен, другой IP после DHCP (если не статика).

### Шаг 6. Прокси (если используется)

Если хост обслуживает **Zabbix proxy**, а не сам server: хост должен быть **назначен** этому прокси (в настройках хоста **Monitored by proxy**), иначе опрос пойдёт не с той машины, и до Arduino трафик не дойдёт так, как вы ожидаете.

### Шаг 7. Триггеры и графики (по желанию)

Это уже стандартный Zabbix: **Triggers** — порог «температура выше X», **Graphs** — визуализация по элементам. На работу прошивки это не влияет.

## Проверка с сервера Linux (утилита `zabbix_get`)

Если на сервере Zabbix установлен пакет с **`zabbix_get`**, с консоли сервера (или прокси), имеющего сеть до Arduino:

```bash
zabbix_get -s <IP_ARDUINO> -p 10050 -k agent.ping
zabbix_get -s <IP_ARDUINO> -p 10050 -k env.temp
```

Ожидается ответ `1` и строка с температурой (например `23.50`). Если тут ошибка — до настройки элементов в GUI нужно устранить сеть/порт/прошивку.

## Формат кадра (сверка с вашим сервером)

Входящий запрос от Zabbix (пассивный агент):

- **13 байт заголовка**: `ZBXD` (4) + версия `0x01` (1) + **длина полезной нагрузки** как **uint64 little-endian** (8).
- Далее **ровно `payloadLen` байт** — ASCII ключа (часто с `\n` в конце; прошивка обрезает `\r`/`\n`).

В прошивке смещение начала ключа задано как **`ZBX_PAYLOAD_OFFSET`** (= 13). Итоговый размер кадра: `13 + payloadLen`.

Ответ:

- `ZBXD` + `0x01` + uint64 LE длины тела + **непрерывные байты** тела (без нулевого терминатора в потоке).

### Как проверить дампом

На машине с Zabbix или рядом с сетью:

```bash
tcpdump -i any -nn -X tcp port 10050 and host <IP_ARDUINO>
```

Убедитесь, что:

1. После `5a 42 58 44 01` восемь байт длины совпадают с длиной хвоста пакета минус 13.
2. Ключ в хвосте начинается с байта **смещения 13** от начала **полезной части TCP** (если один кадр в одном сегменте).

Если ваш сервер шлёт **другую длину заголовка** или смещение ключа не 13 (редкие прокси/обёртки), измените в скетче константы **`ZBX_HEADER_LEN`** и **`ZBX_PAYLOAD_OFFSET`** под фактический размер **до** начала ASCII ключа (сохранив разбор **uint64 LE** в байтах 5–12 относительно начала заголовка протокола Zabbix).

## Ориентиры по железу и стеку

- ENC28J60 + **UIPEthernet**: стабильное питание 3.3 В модуля; в скетче может быть задан статический IP под **10.10.0.0/23** (`255.255.254.0`) — уникальный адрес на каждое устройство в диапазоне **10.10.0.30–10.10.1.254**.
- Альтернатива с большим комьюнити: **W5500** / W5100 и штатная библиотека Ethernet, либо готовый агент **[zabbuino](https://github.com/zbx-sadman/zabbuino)** с множеством команд и тем же стилем протокола.
